# DELTA-V Developer Guide

This guide covers the normal edit, generate, build, and test loop.

Read `docs/ARCHITECTURE.md` first if the runtime layout is unfamiliar.

## Setup

```bash
python3 -m venv .venv
source .venv/bin/activate
pip install -r requirements.txt

cmake -B build -DCMAKE_BUILD_TYPE=Debug
```

## Normal Edit Loop

Use this flow for most changes:

1. Edit code in `src/` or `tests/`.
2. If `topology.yaml` changed, run `python3 tools/autocoder.py`.
3. Rebuild the target you changed.
4. Run the relevant test target.
5. Run `quickstart_10min` or `software_final` before release work.

Common commands:

```bash
cmake --build build --target flight_software
cmake --build build --target run_tests
cmake --build build --target run_system_tests
cmake --build build --target quickstart_10min
```

## Repository Structure

- `src/`: runtime components, hubs, FDIR, and generated headers
- `tests/`: unit and system tests
- `topology.yaml`: source of truth for components, commands, and routing
- `tools/`: generator, validation scripts, and utilities
- `gds/`: Streamlit ground station
- `docs/`: guides, generated reports, and process records

## Creating A Component

Interactive path:

```bash
python3 tools/dv-util.py boot-menu
```

Direct scaffold path:

```bash
python3 tools/dv-util.py quickstart-component ThermalControl --build
```

Manual examples:

```bash
python3 tools/dv-util.py add-component ThermalControl --profile utility
python3 tools/dv-util.py add-component ThermalControl --active --profile controller
python3 tools/dv-util.py add-component ThermalControl --register
```

The optional baseline app bundle can be added with:

```bash
python3 tools/dv-util.py install-standard-apps
```

## Component Rules

Every component should:

- derive from `deltav::Component` or `deltav::ActiveComponent`
- report faults through `recordError()`
- emit events for operator-visible failures
- drain inbound command ports fully during each `step()`
- handle unknown opcodes in `handleCommand()`

## Updating `topology.yaml`

`topology.yaml` is the source of truth for:

- components
- IDs
- commands
- telemetry routes
- event routes
- command classes

Example:

```yaml
components:
  - id: 400
    name: "ThermalControl"
    class: "ThermalControlComponent"
    instance: "thermal"
    type: "Passive"

commands:
  - name: "SET_HEATER_TARGET"
    target_id: 400
    opcode: 1
    description: "Set target temperature in degrees C"
```

After every topology change:

```bash
python3 tools/autocoder.py
python3 tools/autocoder.py --check
```

Generated files:

- `src/Types.hpp`
- `src/TopologyManager.hpp`
- `dictionary.json`

Do not edit generated files by hand.

## Command Classes

Command policy is generated from `topology.yaml`.

Example:

```yaml
commands:
  - name: "MY_NEW_COMMAND"
    target_id: 200
    opcode: 50
    op_class: "RESTRICTED"
    description: "Example restricted command"
```

Meaning:

- `HOUSEKEEPING`: allowed in `BOOT`, `NOMINAL`, `DEGRADED`, and `SAFE_MODE`
- `OPERATIONAL`: blocked in `SAFE_MODE` and `EMERGENCY`
- `RESTRICTED`: allowed in `NOMINAL` only

## Tests And Requirements

Important behavior should map to a requirement in `src/Requirements.hpp` and a
test in `tests/unit_tests.cpp` or `tests/system_tests.cpp`.

Example:

```cpp
// Verifies DV-FDIR-01: system enters SAFE_MODE when battery < 5%
TEST(WatchdogComponent, BatterySafeModeThreshold) {
    ...
}
```

Traceability is generated by the release tooling and used by `software_final`.

## Validation Targets

| Target | Command | Purpose |
|---|---|---|
| Generated-file check | `cmake --build build --target autocoder_check` | Verify generated files are current |
| Flight binary | `cmake --build build --target flight_software` | Host/SITL executable |
| Unit tests | `cmake --build build --target run_tests` | GTest suite |
| System tests | `cmake --build build --target run_system_tests` | End-to-end integration checks |
| V&V stress | `cmake --build build --target vnv_stress` | Repeated shuffled test gate |
| Static analysis | `cmake --build build --target tidy` | clang-tidy |
| Blocking static analysis | `cmake --build build --target tidy_safety` | safety-focused clang-tidy profile |
| Benchmark baseline | `cmake --build build --target benchmark_baseline` | Refresh benchmark artifacts |
| Benchmark guard | `cmake --build build --target benchmark_guard` | Enforce performance thresholds |
| SITL smoke | `cmake --build build --target sitl_smoke` | Short startup/fatal-marker check |
| SITL soak | `cmake --build build --target sitl_soak` | Extended runtime stability gate |
| SITL fault campaign | `cmake --build build --target sitl_fault_campaign` | Live UDP fault exercise |
| Portability matrix | `cmake --build build --target portability_matrix` | Configure/build profile sweep |
| Quickstart | `cmake --build build --target quickstart_10min` | Local validation shortcut |
| Flight readiness | `cmake --build build --target flight_readiness` | Legal + tests + safety + traceability |
| Qualification bundle | `cmake --build build --target qualification_bundle` | Qualification artifacts |
| Software final | `cmake --build build --target software_final` | Sync software evidence to `docs/` |
| Release preflight | `cmake --build build --target release_preflight` | Check release blockers |
| Release candidate | `cmake --build build --target release_candidate` | Tagged release pedigree |
| Review bundle | `cmake --build build --target review_bundle` | Curated review package |

Coverage build:

```bash
bash tools/configure_coverage_build.sh
cmake --build build_cov --target coverage_guard
```

## Running The GDS

Start the runtime:

```bash
./build/flight_software
```

Start the GDS in another terminal:

```bash
streamlit run gds/gds_dash.py
```

For serial KISS mode:

```bash
export DELTAV_LINK_MODE=serial_kiss
export DELTAV_SERIAL_PORT=/dev/tty.usbserial-0001
export DELTAV_SERIAL_BAUD=115200
./build/flight_software
```

## Advanced Topics

TMR-protected parameters:

```cpp
#include "TmrStore.hpp"

TmrStore<float> kp_gain{1.5f};
TmrRegistry::getInstance().registerStore("kp_gain", &kp_gain);
float kp = kp_gain.read();
```

Test-only battery fault injection:

```cpp
topology.watchdog.injectBatteryLevel(4.0f);
EXPECT_EQ(topology.watchdog.getMissionState(), MissionState::SAFE_MODE);
```
